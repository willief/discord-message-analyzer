There is absolutely confusion here. This is not just about trusting binaries. however there is trust in those who create code, that’s for sure.

The mix you have is do you want to allow some scammers, but not too many and do you not want to trust any team, but allow all teams (no code signing etc.) and then encourage folk to download from anywhere?

Attestation is a mechanism where any supplier can safely say this code is correct, the node I am speaking to is also correct.

Otherwise it is a majority decision where some folk are allowed to be incorrect, or scammers and there is no trust at all.

I just don’t think the notion of 100% untrusted code on an untrusted network is valid, somebody has to trust somebody somewhere, otherwise it’s a free for all with all kinds of code.

Majority is a good and solid way of trying to ensure code is correct, but attestation is a way of knowing code is correct.

I am not sure the trustless thing is at all real (it’s a nice meme), we don’t see folk safely downloading bitcoin nodes or wallets from any random place, there does have to be an element of trust. That generally means you trust the code does what it says, not because you can prove it, but because the supplier says so and you trust them.

What I am talking about is “knowing” the code does what it says, even when it’s run on other folk computers. A bit like deterministic builds, but of simple functions and not masses of complex code

It’s not about binaries, we all run binaries, it’s more like signing if you like, where signing proves the source. Here attestation also guarantees the function of the code.

It does not stop many teams creating code and competing either, but it would force teams to collaborate and allow each to attest each others code is functionally correct. It’s been impossible for sure and likely still is, who knows?
